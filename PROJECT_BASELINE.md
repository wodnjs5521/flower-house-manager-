## 변경 시 반드시 지킬 원칙

1. 원본 파일은 보존하고 작업본에서만 수정한다.
2. 한 번에 하나의 목적만 작게 변경한다.
3. 기존 다크 테마, 모바일 최대 폭 460px, 하단 5개 탭을 유지한다.
4. 기존 데이터 필드와 `FHM_DATA_V1` 저장 데이터의 하위 호환성을 유지한다.
5. 작업/수확/출하/공유/동기화 흐름을 임의로 제거하거나 이름을 바꾸지 않는다.
6. 변경 전후에 주요 화면과 데이터 저장 동작을 확인한다.
7. HTML과 JSX 중 어느 파일을 주 작업본으로 정할지 합의한 뒤, 필요하면 다른 버전에도 같은 변경을 반영한다.

## 현재 확인된 주의점

- 로그인 비밀번호와 농장 참여 비밀번호가 평문 데이터로 저장된다.
- Firebase 설정값이 소스에 포함되어 있으며 실제 보안은 Firestore 보안 규칙에 좌우된다.
- HTML은 외부 CDN에 의존하므로 오프라인 최초 실행은 불가능하다.
- HTML과 JSX의 저장 방식이 달라 한쪽만 수정하면 동작 차이가 생길 수 있다.
- 추정 엔진 결과는 센서 실측값이 아니라 날씨와 작업 이력을 조합한 추정치다.

## 기본 작업 방향

별도 요청이 없다면 실제 사용 가능한 기준본인 HTML을 먼저 수정하고, 기능이 안정되면 JSX에 동일 변경을 이식한다.

## 변경 이력

### 1단계 — 오늘 할 일 요약 설명 (2026-06-21)

- `긴급`, `주의`, `절화 필요`, `농약 확인`, `출하 대기` 칸을 누를 수 있는 버튼으로 변경

- 현재 해당되는 하우스 또는 출하 기록을 모달에 표시
- 해당 항목이 0일 때도 설명을 열어볼 수 있음
- HTML과 JSX 작업본에 동일하게 반영

### 2단계 — 출하 상태와 예정 출하일 (2026-06-21)

- `포장 완료` 선택 시 포장 후 상태를 `출하 대기`, `출하 완료`, `출하 보류` 중에서 선택
- 최종 상태가 `출하 대기` 또는 `출하 보류`이면 예정 출하일을 필수 입력
- `출하 완료`는 현재 시점에 완료된 기록으로 저장
- 출하 기록 카드에 포장 완료 여부와 예정 출하일 표시
- 오늘 할 일 요약의 출하 대기 설명에도 예정일 표시
- 기존 출하 기록과 저장 데이터 구조는 그대로 호환

### 3단계 — 수확·출하 탭 분리 및 출하 기록 수정 (2026-06-21)

- 하단 `수확·출하` 탭을 `수확`과 `출하` 탭으로 분리
- 수확 탭에는 하우스별 절화·수확 진행률과 수확 입력만 표시
- 출하 탭에는 출하 등록, 일정, 포장 및 박스 기록만 표시
- 각 출하 기록에 `수정` 버튼 추가
- 출하처, 운송 방식, 상태, 예정일, 꽃·품종, 박스 수와 단 수 수정 가능
- 상세 박스 기록은 기존 박스별 구성을 그대로 불러와 수정
- 수정 시 기존 기록 ID와 최초 등록 시각을 유지하고 `updatedAt`만 추가
- 기존 출하 기록과 저장 데이터 구조는 그대로 호환

### 4단계 — 누적 데이터 기반 AI 개화 예상일 (2026-06-21)

- 정식일과 꽃 이름을 기준으로 실제 연·월·일 개화 범위 계산

- 알려지지 않은 꽃은 기타 화훼 기본 범위로 계산
- 농장 좌표가 있으면 과거 일별 기온, 강수, 습도, 증발산량을 정식일부터 누적
- 사용자의 관수 기록과 날씨의 건조 수요를 함께 반영해 예상일 자동 보정
- 날씨·관수 데이터가 없으면 정식일과 꽃별 기본 생육기간만 사용
- 오늘, 수확, 하우스 상세 화면에 달력 날짜로 표시
- 하우스 상세에 예상 범위, 중심일, 신뢰도, 사용 데이터와 보정 일수 표시

- 꽃별 개화 중심일을 먼저 계산한 뒤 예상 범위를 앞뒤 3일, 총 7일로 제한
- 꽃별 절화 시점 차이를 적용해 AI 절화 중심일을 별도로 계산
- 절화 예상 범위도 중심일 앞뒤 3일, 총 7일로 제한

### 9단계 — 차수별 정식일과 개화 일정 (2026-06-21)

- 하우스 편집에서 정식일을 `1차`, `2차`, `3차` 순서로 추가·삭제
- 기존 단일 정식일은 자동으로 1차 정식일로 호환
- 각 정식일을 기준으로 차수별 개화 중심일과 7일 예상 범위 계산
- 오늘 하우스 상태에는 현재 차수 예상일 하나만 표시
- 1차 중심 예상일이 지나고 2차가 등록돼 있으면 2차 예상일로 자동 전환
- 추가 차수가 있을 때만 핑크색 `+` 버튼 표시
- `+` 일정창에는 사용자가 등록한 2차 이후 일정만 표시하며, 2차까지만 등록했으면 2차만 표시
- 하우스 상세 상단에 1차 정식일을 연·월·일로 표시

### 40단계 — 보안(비밀번호 해시·Firebase 규칙)·오프라인 대비·잔글자 마감 (2026-06-21)

**(가) 비밀번호 평문 저장 → 해시 저장**

- `hashPwd`/`verifyPwd` 추가: 보안 컨텍스트(https·localhost 등)에서는 PBKDF2-SHA256(crypto.subtle, 10만 회), 일부 file:// 같은 비보안 컨텍스트에서는 경량 폴백 해시를 사용합니다. 해시 문자열은 `fhm1$p$salt$hash`(또는 폴백 `fhm1$f$…`) 형식.
- 기존 평문 비밀번호도 그대로 로그인되며(레거시 호환), 로그인에 성공하면 평문을 해시로 자동 업그레이드합니다(소유 농장의 참여 비밀번호 포함).
- 가입·로그인·농장 참여·수확 100% 완료·하우스 삭제 5곳을 비동기 `verifyPwd`로 전환.
- 자동 동기화로 공유되는 농장 참여 비밀번호(joinPassword)도 해시로 저장되므로, Firestore 문서를 읽어도 평문 비밀번호가 노출되지 않습니다.

**(나) 오프라인 최초 실행 불가 → 라이브러리 로컬 동봉**

- React·ReactDOM·Babel을 `vendor/` 폴더에 내려받아 상대경로로 참조합니다. 인터넷 없이도 첫 실행이 됩니다(이 폴더를 index.html과 함께 보관/업로드).
- Firebase는 선택 기능이라 CDN 로드 + 실패 시 graceful 폴백을 유지합니다. 날씨·꽃 사진·동기화는 본래 온라인 기능.
- 참고: 인브라우저 Babel 컴파일은 그대로라 초기 로딩이 다소 느립니다. 빌드 도구가 생기면 JSX 사전 컴파일(Babel 제거) 권장.

**(다) Firebase 키 노출 → 규칙으로 보호 + 문서화**

- 클라이언트 `apiKey`는 비밀이 아니라 식별자이므로 제거 대상이 아닙니다(숨길 수 없음). 실제 보안은 Firestore 규칙이 결정함을 소스 주석·README에 명시.
- 권장 `firestore.rules` 동봉: `farms/{code}` 문서만 허용(코드 형식 검사), 저장 구조(payload 문자열+updatedAt 숫자)·문서 크기(<900KB) 강제, 그 외 경로 전면 차단. 더 강한 보안이 필요하면 Firebase Auth 도입 안내(별도 작업).

**(라) 잔글자 마감·두둑 차수 클램프**

- 깊은 화면(병해충 상세·두둑 편집 등)의 11~12px 잔글자를 일괄 상향(11→12.5, 12→13).
- 정식 차수 삭제 후에도 두둑 칸이 없는 차수(예: 3차)를 표시하지 않도록, 표시 차수를 등록된 차수 수로 클램프.

**검증(브라우저 실측)**

- 신규 회원가입: 가입 즉시 로그인 + 비밀번호·농장 참여 비밀번호가 PBKDF2 해시로 저장(`fhm1$p$…`), 하우스 자동 생성.
- 레거시 평문 계정: 로그인 성공 + 평문→PBKDF2 자동 업그레이드(사용자·농장 참여 비밀번호 모두). 로그아웃 후 해시 검증 경로로 재로그인 성공, 틀린 비밀번호 거부.
- 오프라인 동봉: 스크립트가 `vendor/`에서 로드되어 React 18.3.1 정상 구동, 콘솔 오류 없음.
- HTML·JSX·index.html 괄호 균형, 저장 계층 분리(JSX 스냅샷 / HTML localStorage) 유지 확인.
